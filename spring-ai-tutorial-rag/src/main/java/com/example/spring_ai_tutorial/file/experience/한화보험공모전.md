      # ✅ 조립식 보험 설계 경험 (고민 → 해결 → 성장)

      ## 1. 문제 상황: “보험을 어떻게 DB로 표현하지?”

      초기에는 보험 상품을 단순하게

        - 상품 1개
        - 그에 속한 담보들 (1:N 구조)

      로 설계했습니다.

      하지만 구현을 진행하면서 다음과 같은 문제가 발생했습니다.

        - 동일한 담보를 여러 상품에서 재사용할 수 없음
        - 상품마다 가격/보장금액이 달라야 하는데 구조적으로 불가능
        - “조립식 보험”이라는 기획을 DB가 표현하지 못함

      👉 즉,

      > **비즈니스 로직(조립형 구조)이 DB에 반영되지 않는 문제**를 마주했습니다.
      >
        
      ---

      ## 2. 핵심 고민: “보험은 상품이 아니라 조합이다”

      고민 끝에 내린 결론은 하나였습니다.

      > **보험은 완성된 상품이 아니라 ‘담보의 조합’이다**
      >

      이 관점 전환이 설계를 완전히 바꿨습니다.
        
      ---

      ## 3. 해결: M:N 구조 + Link 테이블 도입

      기존:

        - Product → Coverage (1:N)

      변경:

        - Product ↔ Coverage (M:N)
        - 중간에 **ProductCoverageLink 테이블 도입**

      ### 🔹 핵심 설계 포인트

        - `Coverage` → “재료 (마스터 데이터)”
        - `Product` → “상품 (조합 결과)”
        - `Link` → “상품별 조건 정의”

      Link 테이블에:

        - 가격 (`monthly_price`)
        - 보장금액 (`amount`)
        - 필수 여부 (`is_mandatory`)

      를 넣었습니다.
        
      ---

      ## 4. 이 설계로 해결된 문제

      ### ✅ 1) 담보 재사용 가능

        - “골절 진단비” 하나로
            - 저가 보험
            - 프리미엄 보험
            - 이벤트 보험

      모두 구성 가능
        
      ---

      ### ✅ 2) 상품별 가격 전략 반영

      같은 담보라도

        - A 상품 → 500원 / 30만원 보장
        - B 상품 → 12,000원 / 1,000만원 보장

      👉 **비즈니스 전략을 DB 레벨에서 표현 가능**
        
      ---

      ### ✅ 3) 조립형 보험 구현 가능

        - Core (필수 담보)
        - Rider (선택 특약)

      구분을 `is_mandatory`로 처리
        
      ---

      ## 5. 추가 고민: “비즈니스 규칙을 어디까지 DB로 표현할 것인가?”

      여기서 한 번 더 고민이 생겼습니다.

      > “Core / Rider / 카테고리 규칙을 DB에 다 넣을까?”
      >

      결론:

        - **구조 (관계, 재사용성) → DB**
        - **정책 (검증 로직) → Service**

      로 분리했습니다.
        
      ---

      ## 6. 서비스 레벨에서 해결한 부분

      ### 🔹 조립형 보험 (CUSTOM)

        - Core 담보 **최소 1개 선택 필수**

      ### 🔹 미니 보험 (MINI)

        - Core 담보 **전체 선택 필수**

      👉 DB는 유연하게,

      👉 검증은 Service에서 강제
        
      ---

      ## 7. 구현 과정에서의 시행착오

      ### ❗ 1) 처음엔 모든 걸 정규화하려고 함

      → 테이블 너무 많아지고 복잡해짐

      👉 일부는 JSON / 단순 구조로 타협 필요성 이해
        
      ---

      ### ❗ 2) 단순 DB 설계 vs 실제 서비스 괴리

      → 처음 설계는 “데이터 저장” 중심

      👉 이후:

      > “이걸로 실제 서비스 로직이 돌아가야 한다”
      >

      는 관점으로 전환
        
      ---

      ### ❗ 3) 기술 vs 비즈니스 균형

      → 단순 ERD가 아니라

        - 상품 전략
        - 가격 정책
        - 사용자 선택 흐름

      까지 고려해야 함을 깨달음
        
      ---

      ## 8. 결과 (핵심 성과)

        - 조립형 보험 구조를 **DB 레벨에서 유연하게 설계**
        - 상품/담보 재사용 가능한 구조 확보
        - 서비스 레벨 검증과 분리된 **클린 아키텍처 구현**

        ---

      # ✅ 보험 챗봇 구축 경험 (정리본)

      ## 1. 프로젝트 개요

      보험 약관 기반으로 사용자의 보장 여부를 질의응답 형태로 제공하는

      **LLM + RAG 기반 보험 챗봇 시스템을 설계 및 구현**

        - 사용자가 “이거 보험 적용돼?”와 같은 질문을 하면

          → 실제 가입 상품 + 약관 기준으로 답변 제공


        ---
        
        ## 2. 문제 인식
        
        기존 보험 챗봇의 한계:
        
        - 단순 FAQ 수준 (정해진 질문만 응답)
        - 약관이 길고 복잡 (수백 페이지)
        - LLM 단독 사용 시 **환각(Hallucination)** 위험
        
        👉 핵심 문제:
        
        > “사용자 개인의 가입 정보 + 실제 약관 기준으로 정확한 답변을 어떻게 만들 것인가?”
        > 
        
        ---
        
        ## 3. 핵심 설계 (아키텍처)
        
        ### 🔹 1) RAG 기반 구조 도입
        
        - 보험 약관 PDF를 벡터 DB에 저장
        - 질문 시 관련 약관만 검색 후 LLM에 전달
        
        구성:
        
        - **Vector DB (ChromaDB)** → 약관 저장
        - **Embedding 모델** → 한국어 최적화 모델 사용
        - **LLM** → 답변 생성
        
        ---
        
        ### 🔹 2) 개인화 로직 결합 (핵심 차별점)
        
        단순 약관 검색이 아니라:
        
        - Backend에서
            - `member → contract → contract_detail` 조회
            - 사용자가 가입한
                - 상품
                - 특약 (`coverage_id`)
                    
                    추출
                    
        - Vector 검색 시 필터 적용
        
        ```json
        filter: {
          "coverage_id": { "$in": [101, 105, 110] }
        }
        ```
        
        👉 결과:
        
        - **가입한 특약 범위 내에서만 답변 생성**
        - 타 상품/미가입 보장 내용 제거
        
        ---
        
        ## 4. 약관 데이터 처리 (핵심 구현 경험)
        
        ### 🔹 문제
        
        - 보험 약관 PDF = 수백 페이지
        - 그대로 넣으면 검색 품질 ↓
        
        ---
        
        ### 🔹 해결 방법: "특약 단위 분할 + 메타데이터 설계"
        
        ### 1) PDF → 텍스트 변환
        
        - `LlamaParse` 활용
        
        ### 2) 약관 구조 분리
        
        - “특별약관 단위”로 chunk 분리
        
        예:
        
        - 보통약관
        - 화재손해 특약
        - 상해사망 특약
        
        ---
        
        ### 3) 메타데이터 설계 (중요 포인트)
        
        ```json
        {
          "product_id": 10,
          "coverage_id": 101,
          "coverage_name": "화재손해",
          "category": "SPECIAL_TERMS",
          "page_number": 45
        }
        ```
        
        👉 이걸 통해:
        
        - 특정 특약만 검색 가능
        - 출처 페이지 제공 가능
        
        ---
        
        ## 5. 기술 스택 선택 이유
        
        ### 🔹 Vector DB: ChromaDB
        
        - 로컬 기반 (보안)
        - 빠른 프로토타이핑
        - LangChain과 높은 호환성
        
        ---
        
        ### 🔹 Embedding 모델
        
        - `jhgan/ko-sroberta-multitask`
            - 한국어 특화
            - 비용 없음 (로컬 사용)
            - 가성비 우수
        
        비교:
        
        - OpenAI embedding → 성능 좋지만 비용 + 보안 이슈
        - BGE-M3 → 성능 우수하지만 리소스 부담
        
        ---
        
        ### 🔹 PDF 파싱
        
        - LLM 기반 파싱 (LlamaParse)
            - 구조 인식 정확도 높음
            - 약관 분리 자동화 가능
        
        ---
        
        ## 6. 답변 생성 및 안정성 확보
        
        ### 🔹 환각 방지
        
        - 반드시 RAG 기반으로만 답변 생성
        - 근거 없는 응답 차단
        
        ---
        
        ### 🔹 면책 문구 강제 삽입
        
        예:
        
        > “본 답변은 참고용이며, 정확한 보상 여부는 보험사 심사를 통해 확정됩니다.”
        > 
        
        👉 금융 서비스 특성 고려
        
        ---
        
        ### 🔹 개인정보 보호
        
        - 이름 / 주민번호 등
            
            → **Vector DB 저장 금지**
            
        
        👉 보안 및 컴플라이언스 대응
        
        ---
        
        ## 7. 전체 동작 흐름
        
        1. 사용자 질문 입력
        2. Intent 분석
        3. 사용자 가입 정보 조회 (Backend)
        4. Vector DB 검색 (특약 필터 적용)
        5. 약관 + 사용자 정보 기반 LLM 답변 생성
        6. 면책 문구 포함 후 응답
        
        ---
        
        ## 8. 성과 / 배운 점
        
        - 단순 챗봇이 아닌
            
            → **“개인화된 금융 AI 서비스 구조” 설계 경험**

            - 정확성
